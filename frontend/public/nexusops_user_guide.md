# NexusOps 使用指南（精简版）

## 0. 环境说明（重要）
- 当前环境入口： http://10.88.36.61/
- 该环境有目录安全限制：只允许部署到 `/home/metalm/deploy/NexusOps/*` 下（服务器目标路径必须在该目录范围内）。

## 1. 系统定位
- NexusOps 用于统一管理部署实例、配置文件挂载、CI/CD 触发与审计追踪。
- 核心目标：可控、可追溯、可复用的部署流程。

## 2. 快速上手
### 2.1 系统设置
进入「系统设置」完成以下配置：
- 服务器管理：填写服务器地址、SSH 用户、SSH 私钥、部署路径（deploy_path）。
- 仓库管理：配置 Git 仓库地址、分支、（如需要）访问令牌。
- 权限与角色：确保账号具备 deploy:manage、monitor:read 等必要权限。

### 2.2 部署中心（创建实例）
进入「部署中心」点击创建部署实例：
- 实例名称：建议按业务/环境命名。
- 目标服务器：选择部署目标。
- 来源仓库：选择代码来源与分支。
- 同步源目录：仓库内要同步到目标机的目录（例如 ./develop/backend_frontend/）。
- 服务器目标路径：部署落盘目录（需位于服务器部署路径下）。
- 自定义执行脚本：用于同步完成后的启动/重启（例如 docker compose up -d）。

### 2.3 配置文件管理（实例级配置）
进入实例详情的「配置文件管理」：
- 新增配置文件：按相对路径创建（例如 config.toml、.env、conf/app.yaml）。
- 挂载规则：部署时会把配置文件注入到目标路径，使配置与代码解耦。
- 常见做法：只把环境差异放在配置里，代码统一从仓库同步。

### 2.4 触发部署
在实例详情页触发部署：
- 系统会在 CI 中拉取仓库、同步到目标服务器目标路径。
- 如配置了配置注入，会在目标服务器执行配置包注入。
- 最后执行自定义脚本完成启动/重启。

## 3. 服务实时监控
进入实例详情的「服务实时监控」：
- 展示目标路径下 compose 项目分组与容器状态。
- 正常状态仅显示绿色指示灯；异常状态会高亮并显示文字。
- 运行时长显示为极简格式（例如 45m、2h、3d）。

## 4. 审计与排障
进入「审计日志」查看历史记录：
- 可追踪每次部署的时间、状态、目标服务器与关键变量快照（敏感字段已过滤）。
- 部署失败时优先查看日志，确认：源目录是否存在、目标路径权限、配置注入是否成功、启动脚本是否可执行。

## 5. 常见问题
### 5.1 配置包下载失败
- 确认 NexusOps API 地址可从 CI Runner 访问。
- 确认使用的 token 具备权限且未过期。

### 5.2 目标路径校验失败
- 服务器目标路径必须在服务器部署路径（deploy_path）下，建议统一规划工作空间目录。
